## 1. 개요
현재 개발 중인 'Arthur App'(Next.js 기반)을 외부에서 접속하거나 테스트해 볼 수 있는 **무료 호스팅 서비스**를 제안합니다. 특히 비개발자 실무자인 Juno님의 입장에서 관리하기 편하고 확장이 용이한 서비스들을 중심으로 선정하였습니다.

## 2. 추천 서비스 선정
Arthur App은 Next.js 프레임워크를 사용하고 있으므로, 이를 가장 잘 지원하는 서비스를 우선순위로 두었습니다.

### 2.1. Vercel (강력 추천)
Next.js를 만든 회사에서 운영하는 서비스로, 가장 궁합이 좋습니다.
- **장점**: 깃허브(GitHub) 저장소와 연결하면 코드가 업데이트될 때마다 자동으로 배포됩니다. 무료 플랜(Hobby)의 사양이 매우 넉넉합니다.
- **적합성**: Next.js의 모든 기능을 완벽하게 지원하며, 설정이 거의 필요 없을 정도로 쉽습니다.

### 2.2. Supabase (데이터 및 이미지 저장용)
현재 앱이 `db.json` 파일에 데이터를 저장하고 있는데, Vercel 같은 서비스는 파일을 서버에 영구적으로 저장할 수 없습니다. 이때 필요한 것이 외부 데이터베이스입니다.
- **장점**: 무료로 데이터베이스와 이미지 저장소(Storage)를 제공합니다.
- **적합성**: `db.json` 대신 데이터를 안전하게 보관하고, 업로드된 이미지를 주소(URL) 형태로 관리하기에 최적입니다.

### 2.3. Ngrok (단기 테스트용)
호스팅 서비스에 올리기 전, 내 컴퓨터에서 실행 중인 화면을 잠시 누군가에게 보여주고 싶을 때 유용합니다.
- **장점**: 별도의 배포 과정 없이 명령어 하나로 내 컴퓨터를 외부 서버처럼 만들어줍니다.
- **주의**: 내 컴퓨터가 켜져 있을 때만 접속이 가능하므로 임시 확인용으로만 권장합니다.

## 3. 기술적 주의사항 및 고려사항
현재 앱의 구조상 실제 호스팅 서비스에 올릴 때 고려해야 할 핵심 포인트입니다.

- **데이터 휘발성 문제**: Vercel은 '서버리스' 환경이기에 `fs.writeFileSync`와 같은 명령어로 `db.json` 파일 내용을 수정해도 서버를 다시 켤 때 초기화됩니다.
- **이미지 업로드**: `public/uploads` 폴더에 이미지를 저장하는 방식도 마찬가지로 초기화됩니다. 따라서 이미지는 별도의 클라우드 저장소(Supabase Storage 등)를 사용해야 합니다.
- **환경 변수**: `GEMINI_API_KEY` 같은 중요한 정보는 Vercel 설정 화면에서 한 번만 입력해주면 안전하게 관리됩니다.

## 4. 로드맵 제안
Juno님의 생산성을 높이기 위해 다음과 같은 순서로 진행해 보시는 것을 추천합니다.

**1단계: Vercel 배포**
- 깃허브에 코드를 올리고 Vercel과 연결하여 우선 화면이 잘 뜨는지 확인합니다.

**2단계: 데이터 저장소 이전 (옵션)**
- 서비스가 실제 운영 단계로 넘어가야 한다면, `db.json` 대신 **Supabase**를 연동하여 데이터가 사라지지 않게 구성합니다.

> [!tip] 개인적인 추천
> 처음에는 복잡하게 생각하지 마시고 **Vercel**에 한 번 올려보시는 것을 강력 추천합니다. "내 앱이 인터넷 주소로 접속된다"는 경험만으로도 작업의 체감이 크게 달라질 것입니다.
