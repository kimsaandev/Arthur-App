## 1. 개요
안녕하세요, **Juno**입니다. 앞선 보고서에서 다룬 이미지 파일(AI 생성물, 인스타 합성본) 외에도, 서버를 운영하다 보면 **"보이지 않게 용량을 잡아먹는 복병"** 들이 있습니다. 1,000명이 매일 이용하는 활발한 서비스일수록 이 문제들은 더 빨리 실체화됩니다.

이미지 외에 유의미하게 용량이 늘어날 수 있는 **4가지 핵심 요인**을 분석해 드립니다.

## 2. 보이지 않는 용량 증가 요인

### 2.1. 애플리케이션 로그 (Application Logs)
현재 앱 코드 곳곳에는 `console.log()`가 배치되어 있습니다. 사용자 1명이 이미지를 생성할 때마다 서버 터미널에는 작업 로그가 남습니다.
- **예상**: 생성 시마다 발생하는 상세 로그(에러 로그 포함)가 텍스트 파일로 저장될 경우, 5개월간 수십만 라인의 텍스트가 쌓입니다.
- **규모**: 로그 관리 설정(Log Rotation)이 없다면 로그 파일 하나가 수 GB까지 커질 수 있습니다.

### 2.2. Next.js 빌드 및 이미지 캐시 (.next/cache)
Next.js는 성능 향상을 위해 이미지와 페이지 데이터를 캐시(임시 저장)합니다.
- **내용**: 특히 `next/image` 컴포넌트를 사용하여 이미지를 리사이징하거나 최적화할 경우, 그 결과물이 `.next/cache/images` 폴더에 차곡차곡 쌓입니다.
- **영향**: 원본 이미지의 크기가 변할 때마다 새로운 캐시가 생성되므로, 운영 기간이 길어질수록 빌드 폴더의 크기가 수백 MB에서 수 GB 단위로 늘어나게 됩니다.

### 2.3. 데이터베이스 파일 (db.json)
현재 `lowdb`를 사용하여 JSON 파일에 작업 내역을 저장하고 있습니다.
- **리스크 관리**: 다행히 코드 내에 최신 50건만 유지하도록 제한(`slice(-50)`)이 걸려 있어, 이 파일 자체가 무한정 커지지는 않습니다.
- **주의**: 다만, 만약 이 제한을 풀거나 기록 보관을 위해 전체 데이터를 저장하도록 변경한다면, 수십만 건의 텍스트 데이터가 쌓여 파일 읽기 속도가 느려지고 용량도 늘어나게 됩니다.

### 2.4. 임시 파일 및 의존성 (Node Modules & Temp)
- **Sharp 캐시**: 이미지 합성을 담당하는 `sharp` 라이브러리가 내부적으로 캐시를 사용할 수 있습니다.
- **의존성 업데이트**: 운영 중 새로운 라이브러리를 추가하거나 버전을 올리다 보면 `node_modules` 폴더 자체의 크기도 조금씩 늘어납니다. (현재 약 350+ 패키지 설치 중)

## 3. 운영을 위한 종합 제안

단순히 **405GB(이미지)** 만 생각하기보다, 보이지 않는 캐시와 로그를 위해 **약 50~100GB 정도의 추가 여유분**을 고려하는 것이 안전합니다.

> [!tip] 실무자의 조언
> - **로그 로테이션(Log Rotation)**: 로그 파일이 일정 크기가 되면 압축하고 오래된 것은 삭제하는 설정을 꼭 적용하세요.
> - **캐시 초기화**: 서버 배포 시나 주기적으로 `.next/cache`를 비워주는 작업을 관리 루틴에 포함하면 저장 공간 관리가 훨씬 수월해집니다.

추가로 분석이 더 필요한 부분이 있다면 언제든 말씀해 주세요!

[^1]: **로그 로테이션**: 로그 파일의 크기가 무한정 커지는 것을 방지하기 위해 파일 용량이나 날짜를 기준으로 파일을 새로 만들고 관리하는 기법입니다.
[^2]: **캐시 (Cache)**: 자주 사용하는 데이터를 미리 저장해두어 속도를 높이는 임시 저장소입니다.
