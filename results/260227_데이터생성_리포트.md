# 앱 운영 시 데이터 생성 및 저장 경로 분석 리포트

안녕하세요, Juno입니다.

Arthur-App을 직접 서버에 올리고 24시간 내내 운영하다 보면, "이 많은 데이터들이 도대체 내 서버의 어디에, 어떻게 쌓이고 있는 걸까?"라는 궁금증이 생기기 마련입니다.

이미지가 너무 많이 쌓여서 용량이 꽉 차는 건 아닐지, 혹은 중요한 설정 값들이 어디에 보관되는지 모른다면 불안감이 생길 수밖에 없죠. 비개발자 실무자 입장에서도 내 컴퓨터(서버) 안에서 어떤 폴더에 어떤 파일들이 생겨나는지 그 원리를 간단하게 파악해두면, 앱 유지보수가 훨씬 수월해집니다.

이번 글에서는 **이 앱이 구동되고 방문자가 사용할 때, 데이터가 어떻게 생성되고 어디에 저장되는지** 직관적으로 정리해 보았습니다.

---

## 1. 텍스트와 기록이 남는 곳: 데이터베이스 (SQLite)

가장 먼저 우리 앱의 모든 설정과 방문자의 이용 내역이 차곡차곡 쌓이는 장부 역할을 하는 곳입니다. 엑셀 파일 하나에 모든 걸 적어둔다고 생각하시면 이해하기 쉽습니다.

### 1.1. 파일 위치 및 역할
- **위치**: `data/db.sqlite` (그리고 `db.sqlite-wal`, `db.sqlite-shm` 파일들)
- **저장되는 내용**:
  - **방문자 기록 (`tasks` 테이블)**: 누가 언제 접속해서 이미지를 만들었는지, 생성된 이미지의 위치가 어딘지 적혀있습니다.
  - **관리자 설정 (`settings` 테이블)**: 어드민 페이지에서 우리가 입력한 Gemini API Key나 BFL Flux Key 값, 그리고 현재 어떤 AI 모델을 쓰고 있는지가 안전하게 암호화되어 저장됩니다.

> [!note] 꿀팁: SQLite란?
> 보통 데이터베이스라고 하면 거창한 서버를 생각하지만, SQLite는 그냥 **하나의 파일(`db.sqlite`)로 존재하는 간편한 데이터베이스**입니다. 나중에 백업할 때 이 `data` 폴더 전체만 복사해두면 모든 기록이 완벽하게 백업됩니다.

### 1.2. 언제 생성되나요?
- **앱을 처음 켰을 때**: `data` 폴더와 `db.sqlite` 파일이 없다면 자동으로 스스로 뼈대(테이블)를 만듭니다.
- **사용자가 사진을 생성할 때마다**: 장부에 새로운 줄이 틱틱 생겨나듯 기록이 한 줄씩 자동으로 연장됩니다.

---

## 2. 시각적인 결과물이 남는 곳: 이미지 저장소

우리가 이 앱을 돌리는 핵심 이유죠! AI가 합성해서 만들어준 예쁜 결과물들은 파일 형태로 디스크에 영구적으로 보관됩니다.

### 2.1. 폴더 위치 및 역할
- **위치**: `public/uploads/`
- **저장되는 내용**:
  - **원본 결과 이미지**: `20260227_143000_juno.jpg` 처럼 날짜와 시간이 박힌 이미지 파일이 생성됩니다.
  - **인스타용 합성 이미지**: `20260227_143000_insta_juno.jpg` 처럼 인스타그램 템플릿에 딱 맞춰진 별도의 이미지도 함께 스크립트를 통해 자동으로 만들어져 저장됩니다.

> [!warning] 용량 관리 주의!
> `public/uploads/` 폴더는 앱을 오래 켜둘수록 사진 파일들이 계속해서 쌓이는 곳입니다. 몇 달, 몇 년간 운영한다면 용량이 꽉 찰 수 있으니 가끔씩 주기적으로 들어가서 불필요한 예전 사진들은 백업하고 지워주는 관리가 필요합니다. (또는 나중에 오래된 사진 자동 삭제 기능을 추가하는 것을 추천합니다!)

### 2.2. 언제 생성되나요?
- **이미지 생성 완료 시**: AI(Gemini나 Flux)가 합성을 성공적으로 마친 즉시, 화면에 뿌려주기 직전에 이 폴더에 실제 `.jpg` 파일로 저장됩니다.

## 3. 앱의 건강 상태를 보여주는 곳: 로그 (Logs) & 캐시 (Cache)

앱이 잘 돌아가고 있는지, 혹은 언제 청소를 했는지 앱 스스로 다이어리를 쓰는 공간입니다.

### 3.1. 로그 기록 및 유지보수 이력
- **위치**: `logs/last-cleanup.json`, `logs/app.log`
- **역할**: 매일 새벽 4시마다 앱이 깨어나서 스스로 불필요한 임시 파일들을 지우는 청소(Cleanup) 작업을 진행합니다. 이때 "오늘 청소 잘 마쳤어~"라고 기록을 남겨두는 파일입니다.

### 3.2. 프레임워크 임시 저장소 (Next.js Cache)
- **위치**: `.next/cache/`
- **역할**: 앱의 접속 속도를 엄청나게 빠르게 만들기 위해 Next.js라는 엔진이 스스로 만들어두는 **임시 조각 모음**입니다. 캐시가 있으면 페이지 로딩이 빨라지지만, 오류가 생길 땐 여기서 문제가 꼬이기도 합니다.
- 다행히 매일 새벽 4시에 실행되는 청소봇(`scripts/cleanup.js`)이 이 무거워진 캐시 폴더를 알아서 주기적으로 날려주고 다시 만들도록 설계되어 있어 크게 신경 쓰지 않으셔도 괜찮습니다.

## 4. 마치며

정리하자면 우리 서버에서는 크게 **두 가지 바구니**만 잘 관리하면 됩니다!

1. `data/db.sqlite`: 📊 텍스트 기록이 적힌 장부 (가벼우나 삭제되면 안됨)
2. `public/uploads/`: 🖼️ AI가 만든 이미지 파일 보관소 (무거워질 수 있으니 용량 관리가 필요함)

나중에 서버를 이사하거나 만약의 사태에 대비해 수동으로 백업을 챙겨야 한다면, 이 두 가지 폴더(`data/` 와 `public/uploads/`)만 압축해서 가지고 계시면 전체 앱의 모든 데이터를 완벽하게 복구할 수 있습니다. 생각보다 아주 직관적이고 심플하게 구조가 잡혀있죠?

앞으로 앱을 쾌적하게 운영하시는 데에 이 데이터 파이프라인 정리본이 유용한 나침반이 되기를 바랍니다!
