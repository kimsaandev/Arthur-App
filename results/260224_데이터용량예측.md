## 1. 개요
안녕하세요, **Juno**입니다. Arthur App의 본격적인 운영을 앞두고 가장 고민되실 **서버 저장 공간(스토리지) 문제**에 대한 분석 결과를 준비했습니다. 매일 1,000명의 사용자가 유입된다는 것은 긍정적인 신호지만, 그만큼 무거운 이미지 데이터가 서버에 쌓인다는 뜻이기도 합니다.

이 보고서에서는 현재 앱에서 실제로 생성되고 있는 샘플 이미지의 크기를 바탕으로, **5개월 간 운영할 때 어느 정도의 데이터가 쌓일지** 예측하고 그에 따른 대비책을 제안합니다.

## 2. 데이터 용량 산출 기준

현재 개발 환경(`public/uploads/`)에 저장된 파일들을 직접 열어보고 평균적인 용량을 측정했습니다.

- **AI 생성 이미지 (원본)**: 약 850KB ~ 950KB
- **인스타그램 공유용 합성 이미지**: 약 900KB
- **로직 특성**: 사용자가 1회 생성 버튼을 누를 때마다 **1개의 원본 이미지**와 **1개의 인스타용 이미지**가 동시에 서버에 저장됩니다.

> [!info] 1회 생성 시 필요 용량
> AI 원본(0.9MB) + 인스타 합성(0.9MB) = **총 1.8MB**가 사용됩니다.

## 3. 5개월 누적 데이터 예측 시뮬레이션

제시해주신 트래픽 지표(일 방문자 1,000명, 인당 1.5회 생성)를 바탕으로 데이터 증가량을 일간, 월간, 5개월 누적으로 나누어 계산했습니다.

### 3.1. 트래픽 및 용량 계산표

| 구분 | 계산식 (수식) | 예상 생성 건수 | 예상 데이터 크기 |
| :--- | :--- | :--- | :--- |
| **일간 (1일)** | 1,000명 × 1.5회 | 1,500건 | **약 2.7 GB** (1500 × 1.8MB) |
| **월간 (30일)** | 1,500건 × 30일 | 45,000건 | **약 81 GB** |
| **운영 (5개월)** | 45,000건 × 5개월 | 225,000건 | **약 405 GB** |

### 3.2. 실무 환경에서의 필요 용량
수치상으로는 405GB가 필요하지만, 실제 서버 운영 시에는 운영체제(OS) 설치 공간, 앱 구동 파일들, 그리고 예기치 못한 트래픽 급증을 대비한 여유 공간이 필요합니다.

> [!warning] 주의사항
> 스토리지 공간이 꽉 차게 되면 앱이 완전히 멈춰버리는(다운되는) 치명적인 오류가 발생합니다. 따라서 실제 필요량보다 **20~30% 여유**를 두는 것이 안전합니다.

## 4. 마치며 및 해결 제안

분석 결과, 5개월을 안정적으로 운영하기 위해서는 **최소 500GB 이상의 넉넉한 스토리지**를 확보하는 것이 좋습니다.

하지만 시간이 지날수록 무한정 용량을 늘릴 수는 없기 때문에 아래와 같은 대안을 고려해 보시는 것을 강력히 추천합니다.

1. **클라우드 스토리지 도입 (AWS S3 등)**: 앱이 켜져 있는 서버 대신, 사진만 전문적으로 보관하는 아주 저렴하고 무제한인 클라우드 창고를 빌려 쓰는 방식입니다.
2. **자동 삭제 주기 설정 (Retention)**: 유저가 갤러리에서 사진을 다운로드할 충분한 시간(예: 생성 후 7일)이 지나면 영구 삭제하여 공간을 재활용하는 방식입니다.

앱의 확장 계획에 따라 어떤 솔루션이 가장 유리할지 추가로 논의해 보아도 좋을 것 같습니다. 보고서 내용 중 이해가 안 가시는 부분이나 추가적인 궁금증이 있다면 언제든 편하게 질문해 주세요!

[^1]: **트래픽 (Traffic)**: 서버에 접속하는 사람의 수나 주고받는 데이터의 총량을 의미합니다.
[^2]: **AWS S3**: 아마존에서 제공하는 파일 보관 서비스로, 쓴 만큼만 비용을 내기 때문에 스타트업에서 가장 많이 사용하는 창고입니다.
