## 1. 개요
나스A에서 테스트 중 수집된 보완점들을 반영하여 로컬에서 앱을 개선한 후, 이를 다시 운영 중인 나스A에 안전하고 빠르게 적용하는 **'지속적 개선 및 배포 워크플로우'**를 안내합니다.

## 2. 표준 업데이트 워크플로우 (Local to NAS)

개선된 기능을 나스에 적용하는 과정은 크게 3단계로 이루어집니다.

### 1단계: 로컬 개선 및 검증 (Local Development)
- **수정**: 현재 저(Antigravity)와 함께 진행하시는 것처럼 보완사항을 코드로 수정합니다.
- **검증**: `npm run dev`로 로컬에서 기능이 의도대로 작동하는지 최종 확인합니다.
- **푸시**: 변경된 코드를 GitHub 저장소로 올립니다(`git push`).

### 2단계: 나스 데이터 동기화 (Sync on NAS)
나스의 SSH 터미널 또는 작업 스케줄러를 통해 최신 코드를 내려받습니다.
```bash
# 나스의 프로젝트 폴더로 이동 후
git pull origin main
```

### 3단계: 컨테이너 재빌드 및 재시작 (Rebuild)
수정된 코드를 바탕으로 도커 이미지를 다시 만들고 서비스를 재시작합니다.
```bash
docker-compose down
docker-compose up -d --build
```

## 3. 핵심 주의사항: 데이터 보존 (Data Persistence)

업데이트 중에 그동안 쌓인 소중한 사진과 데이터베이스가 사라지지 않도록 하는 것이 가장 중요합니다.

- **볼륨 매핑(Volumes)**: 현재 `docker-compose.yml` 설정상 `public/uploads`와 `db.sqlite` 등은 나스의 실제 하드디스크 폴더와 연결되어 있습니다.
- **안전성**: 따라서 컨테이너를 삭제하고 다시 만들어도(Rebuild), 나스 폴더에 저장된 **사진 데이터와 DB는 그대로 유지**됩니다.
- **주의**: 새로 추가되는 파일이나 폴더가 있다면 `docker-compose.yml`의 `volumes` 섹션에 미리 등록되어 있는지 확인해야 합니다.

## 4. 발생 가능한 문제 및 대응 (Troubleshooting)

| 예상 문제 | 원인 | 대응 방안 |
| :--- | :--- | :--- |
| **빌드 오류** | 나스 사양 부족 또는 네트워크 불안정 | `npm install` 과정에서 멈춘다면 나스의 리소스 모니터를 확인하고 재시도합니다. |
| **데이터 유실** | 볼륨 설정 오기입 | 새 버전 배포 전 반드시 `db.sqlite` 파일을 별도 장소에 백업해 둡니다. |
| **권한 문제** | 새 폴더 생성 시 권한 미부여 | 업데이트 후 이미지 업로드가 안 된다면 `public/uploads` 권한을 다시 체크합니다. |

## 5. 결론 및 제언

> [!tip] **효율적인 운영 팁**
