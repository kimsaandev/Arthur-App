## 1. 개요
현재의 `db.json` 방식은 소규모 데이터에는 간편하지만, 수십만 건의 데이터를 다루는 5개월간의 전시회 운영에는 한계가 있습니다. 이를 해결하기 위해 **SQLite** 데이터베이스로 전환할 때 필요한 요건과 예상되는 제약사항을 분석했습니다.

## 2. 도입을 위한 기술적 요건

### 2.1. 필요 라이브러리 (Dependencies)
- **`better-sqlite3` 또는 `sqlite3`**: Node.js에서 SQLite를 다루기 위한 핵심 라이브러리입니다. (`better-sqlite3`가 성능과 동기식 API 지원 측면에서 더 유리합니다.)
- **ORM (Prisma 또는 Kysely) 선택 사항**: 데이터베이스 작업을 더 쉽고 안전하게 하기 위해 사용할 수 있으나, 가벼운 앱인 만큼 직접 SQL을 사용하는 것도 좋은 방법입니다.

### 2.2. 코드 수정 범위
- **Data Access Layer**: `saveToDb` 함수와 `api/tasks` 내의 JSON 읽기/쓰기 로직을 SQL 쿼리(`INSERT`, `SELECT`, `DELETE`)로 교체해야 합니다.
- **Initialization**: 앱 시작 시 `db.sqlite` 파일이 없으면 자동으로 테이블을 생성하는 로직이 필요합니다.

## 3. 시나리오별 제약 및 주의사항

### 3.1. 시놀로지 나스(Docker) 환경의 특수성
- **파일 시스템 잠금 (File Locking)**: SQLite는 파일 기반 DB입니다. 나스의 네트워크 드라이브(NFS, SMB)를 통해 볼륨을 연결할 경우, 파일 잠금 메커니즘이 제대로 작동하지 않아 DB가 손상될 위험이 있습니다. 
- **[해결책]**: 반드시 나스의 **내부 로컬 볼륨(SSD/HDD)** 내의 폴더를 도커와 직접 매핑해야 합니다.

### 3.2. 동시성 제약 (Concurrency)
- **쓰기 잠금**: SQLite는 한 번에 하나의 프로세스만 데이터를 쓸 수 있습니다. 수백 명의 사용자가 **동시에** '생성' 버튼을 누를 경우 미세한 대기 시간이 발생할 수 있습니다. 
- **[분석]**: 1,000명 규모의 일일 트래픽이라면 SQLite의 성능으로도 충분히 여유롭게 처리 가능합니다.

## 4. SQLite 전환 시 얻게 될 변화

| 항목 | 현재 (JSON) | 변경 (SQLite) |
| :--- | :--- | :--- |
| **속도** | 데이터가 많아질수록 느려짐 | 데이터 양에 상관없이 빠름 |
| **보안** | 파일 전체가 노출됨 | 데이터 검색 및 접근 제어 용이 |
| **확장성** | 50개 제한 필수 | 수십만 건 보관 가능 |
| **유지보수** | 수동 아카이빙 필요 | 인덱스(Index) 관리로 자동화 |

## 5. 결론 및 제안
SQLite 전환은 **"5개월 무중단 운영"**을 위해 매우 현명한 투자입니다. 특히 나스 도커 환경에서 운영하신다면, 볼륨 매핑 설정만 주의하면 클라우드 부럽지 않은 성능을 낼 수 있습니다.

> [!important] 중요
> SQLite로 전환하면 기존 `db.json`에 있는 데이터는 한 번 마이그레이션(이전) 작업이 필요합니다.

이후 구체적인 변환 작업을 진행하고 싶으시다면, 실제 코드 수정안과 패키지 설치를 도와드리겠습니다.

[^1]: **인덱스(Index)**: 책의 색인처럼, 특정 데이터를 빠르게 찾을 수 있도록 돕는 데이터베이스 내부 지도입니다.
[^2]: **볼륨 매핑**: 컨테이너 내부의 저장소와 실제 나스의 저장 폴더를 연결하는 설정입니다.
