# Arthur-App 보안 취약점 점검 보고서

안녕하세요! 요청하신 내용에 따라 서버의 인젝션 방어 수준과 `gallery`, `result`, `admin` 등 주요 페이지의 무단 접속 가능 여부를 상세히 점검했습니다. 앱을 더 안전하고 튼튼하게 만들기 위해 제가 확인한 내용과 권장 사항을 꼼꼼하게 정리해 보았습니다.

---

## 1. 데이터 입력 (인젝션 및 DoS) 보안 점검

서버가 다운되거나 정보가 유출될 수 있는 '입력 폼(이름 자리와 사진 업로드)'의 보안을 확인한 결과, **방어 대책이 매우 잘 마련되어 있습니다.**

### 1.1. 사진 업로드 용량 제한 (DoS 방어)
사진 데이터를 받을 때 약 15MB 이상의 악의적인 거대 파일(Base64 문자열)이 들어오면 서버가 과부하되어 뻗는 것을 막기 위해, 코드 상에 확실한 길이 제한 방어막이 쳐져 있습니다.
- `if (userImage.length > 15000000) { ... }` 
> [!check] 
> 너무 큰 이미지는 사전에 차단되므로 누군가 서버 용량을 고갈시킬 목적으로 폭탄을 던지는 공격(DoS)은 안전하게 방어되고 있습니다.

### 1.2. 이름 입력란 (인젝션 및 경로 탐색 방어)
입력된 이름은 파일 저장 시 파일명으로 사용되는데, 여기서 흔히 발생하는 "경로 탐색(Directory Traversal)" 공격이나 "XSS(교차 사이트 스크립팅)" 공격을 막기 위해 철저한 필터링이 적용되어 있습니다.
- 영문, 숫자, 한글 이외의 모든 특수문자를 언더바(`_`)로 치환합니다.
- 파일명 뒤에는 항상 `.jpg`를 강제로 붙이도록 설계되어 있어, 해커가 악성 스크립트 파일(`.php`, `.html` 등)을 몰래 업로드하고 실행하는 것을 원천 차단합니다.
> [!note] 
> 특수문자 입력 자체가 불가능하게 설계되어 악의적인 코드가 주입(Injection)되거나 다른 폴더의 파일 정보가 탈취될 위험성은 거의 없습니다.

### 1.3. 데이터베이스 쿼리 (SQL 인젝션 방어)
데이터베이스(SQLite)에 기록할 때는 `stmt.run(id, imageUrl, ...)` 형태의 안전한 바인딩 방식을 사용하고 있습니다. 
> [!tip]
> 이 방식은 값을 단순한 '문자'로만 취급하기 때문에 데이터베이스를 탈취하려는 SQL 인젝션 공격이 시도되더라도 전혀 작동하지 않습니다.

## 2. 비밀 페이지 접근 권한 점검

`gallery`(갤러리), `result`(결과 화면), `admin`(관리자) 페이지들이 외부에서 몰래 접속 가능한지 확인해보았습니다. 결론부터 말씀드리면, **해당 페이지 주소를 아는 사람이라면 누구나 접속이 가능합니다.**

### 2.1. 미들웨어(Middleware) 보안 예외 처리
현재 우리 앱의 중앙 통제소 역할을 하는 `src/middleware.ts` 파일을 보면, QR코드를 통한 쿠키를 발급받아야만 메인 페이지 등을 쓸 수 있도록 안전하게 보호 중입니다. 
하지만 다음 페이지들은 통제소의 감시망에서 **예외 처리**되어 있습니다.
- `'/((?!...|result|gallery|admin|...).*)'`
> [!warning] 주의할 점
> 예외 처리가 되어 있기 때문에 주소창에 `인터넷주소/gallery` 또는 `인터넷주소/result`라고 직접 타이핑하면 누구라도 인증 없이 바로 페이지가 열립니다. 특히 해당 이름들은 직관적이어서 쉽게 유추가 가능합니다.

### 2.2. 페이지별 실제 접근 여부
- **`gallery` (갤러리 페이지)**: 
  - 누구나 접속해서 사람들의 합성 결과물을 구경하고 다운로드할 수 있습니다. 
  - 다행히 사진을 **삭제**할 때는 비밀번호(`7575`)를 요구하므로 권한 없는 삭제는 불가능합니다.
- **`result` (결과 송출 페이지)**: 
  - 역시 인증 없이 접근 가능합니다. 누군가 주소를 알아내어 띄워놓는다면 현재 송출되고 있는 이미지를 볼 수는 있습니다.
- **`admin` (관리자 페이지)**:
  - 주소로 접근하면 페이지 자체는 열리지만, 접속 전 "7575admin"이라는 비밀번호를 입력해야 시스템 설정 메뉴로 진입할 수 있게 **자체적인 1차 잠금장치**가 걸려 있습니다.
  - 설정을 덮어쓰거나(POST) 변경할 때에도 백엔드 API 단에서 비밀번호를 한 번 더 체크하고 있어 API 키 유출 위험은 적습니다.

## 3. 마치며 (요약 및 권장 사항)

현재 서버를 터뜨리거나 악성 코드를 주입하는 "치명적인 해킹 공격"에 대해서는 시스템이 아주 잘 방어하고 있어 크게 걱정하지 않으셔도 괜찮습니다. 제법 든든한 상태입니다!

다만, 갤러리나 결과 페이지가 의도하지 않은 사람에게 노출되는 것이 불편하시다면 아래와 같은 후속 조치를 고려해 볼 수 있습니다.

1. **페이지 이름 단순 변경 (가장 쉽고 빠른 방법)**
   - 페이지 폴더명을 유추하기 힘든 복잡한 이름(예: `x9_gallery_secret`)으로 변경하면 몰래 접속하려 해도 주소를 찾기 매우 어려워집니다.
2. **비밀번호 잠금 처리 (권장)**
   - 관리자 페이지에 적용된 것처럼, 갤러리와 결과 페이지에도 간단한 통과 비밀번호를 묻는 화면을 먼저 띄운 뒤에만 내용이 보이도록 수정할 수 있습니다.

이 중 조치를 원하시는 방향이 있다면 언제든지 말씀해 주세요. 바로 적용해 드리겠습니다!
