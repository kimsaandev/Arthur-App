## 1. 개요
안녕하세요, **Juno**입니다. Vercel이나 Firebase 같은 클라우드 서비스 대신, 개인 또는 사무실에서 운영 중인 **시놀로지 나스(Synology NAS)** 를 활용해 앱을 서비스하는 방안을 분석했습니다. 결론부터 말씀드리면, Arthur App은 도커(Docker) 기술을 활용하기에 매우 최적화된 구조를 가지고 있어 **나스 운영이 충분히 가능하며 권장할 만한 선택지**입니다.

## 2. 시놀로지 나스 운영의 장단점 분석

나스에서 직접 운영할 때 얻을 수 있는 실질적인 이점과 고려해야 할 사항을 정리했습니다.

### 2.1. 주요 장점
- **비용 절감**: 앞서 분석한 대로 5개월간 400GB 이상의 데이터가 쌓일 경우, 유료 클라우드 서비스는 상당한 스토리지 비용이 발생합니다. 하지만 나스는 이미 보유한 하드디스크 공간을 사용하므로 **추가 비용이 0원**에 가깝습니다.
- **데이터 보안 및 프라이빗 운영**: AI로 생성된 이미지와 방문객의 사진이 외부 서버에 남지 않고, 우리 사무실 안에 있는 나스에만 저장되므로 개인정보 보호 측면에서 매우 안전합니다.
- **성능 및 안정성**: 나스의 하드웨어 성능이 충분하다면(보통 + 모델 이상), 1,000명 정도의 트래픽은 안정적으로 처리할 수 있습니다.

### 2.2. 고려해야 할 사항 (Checklist)
- **외부 접속 설정**: 나스는 기본적으로 내부망에 있으므로, 전시회 현장에서 접속하려면 **포트 포워딩(Port Forwarding)** 이나 **DDNS** 설정이 필요합니다.
- **하드웨어 사양**: 도커를 원활히 돌리기 위해서는 Intel/AMD 프로세서가 탑재된 **'+' 시리즈 모델(예: DS920+, DS723+ 등)** 과 충분한 RAM(최소 4GB 이상)이 권장됩니다.

## 3. 도커(Docker) 기반 설치 전략

나스의 `Container Manager`(구 Docker 패키지)를 사용하여 다음과 같이 구성할 수 있습니다.

### 3.1. 환경 구성 요약
- **이미지 생성**: 소스 코드를 도커 이미지로 빌드하여 나스에 업로드합니다.
- **볼륨 매핑 (중요)**: 앱 내부의 `db.json`, `public/uploads`, `logs` 폴더를 나스의 실제 폴더와 연결합니다. 이렇게 하면 도커 컨테이너를 삭제하거나 업데이트해도 **우리의 소중한 데이터(이미지, 로그)는 안전하게 보존**됩니다.

## 4. 마치며
시놀로지 나스 운영은 초기 설정이 클라우드보다 조금 까다로울 수 있지만, **운영 비용과 데이터 주권** 측면에서 압도적인 우위에 있습니다. 특히 이번 프로젝트처럼 대용량 이미지가 발생하는 경우에는 나스 운영을 적극 추천합니다.

설치를 직접 진행하실 수 있도록 구체적인 **Dockerfile**과 **Docker Compose** 설정 파일도 함께 준비해 두었으니 필요할 때 언제든 말씀해 주세요!

[^1]: **도커(Docker)**: 앱을 실행에 필요한 모든 환경과 함께 통째로 가상화된 상자에 담아 어디서든 똑같이 실행할 수 있게 해주는 기술입니다.
[^2]: **DDNS**: 유동 IP를 가진 나스에 `juno-arthur.synology.me` 같은 고정된 주소를 부여해주는 서비스입니다.
